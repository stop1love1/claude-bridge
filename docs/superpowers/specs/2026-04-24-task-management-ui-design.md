# Task Management UI — Design Spec

**Date:** 2026-04-24
**Project:** edusoft-lms-bridge
**Author:** Owner/coordinator (solo)

## Goal

A lightweight local web UI that lets the bridge owner:

1. Create and assign tasks (one central place; no manual markdown edits).
2. Auto-dispatch each task to an agent team (coordinator → coder → reviewer) that executes in the right sibling app (`edusoft-lms` / `edusoft-lms-api`).
3. Browse live and historical Claude Code sessions (native JSONL) per task.

The bridge remains the canonical **owner/coordinator** of the system; the UI is its front door.

## Non-goals

- Multi-user / auth / sharing — solo tool, local only.
- Replacing `tasks.md` as source of truth — UI reads/writes it.
- Custom session format — sessions are read from `~/.claude/projects/<slug>/*.jsonl` exactly as Claude Code writes them.
- CI / remote execution / scheduling — run-on-demand only.

## Architecture

```
edusoft-lms-bridge/

│   ├── lib/
│   │   ├── tasks.ts           parse/write tasks.md
│   │   ├── sessions.ts        list/tail ~/.claude/projects/<slug>/*.jsonl
│   │   ├── meta.ts            read/write sessions/<task-id>/meta.json
│   │   └── spawn.ts           child_process.spawn wrapper for `claude`
│   └── public/
│       ├── index.html         single-page UI
│       ├── app.js             vanilla JS
│       └── styles.css
├── agents/
│   ├── coordinator.md         prompt template (role=coordinator)
│   ├── coder.md               prompt template (role=coder)
│   └── reviewer.md            prompt template (role=reviewer)
├── sessions/
│   └── <task-id>/
│       └── meta.json          {taskId, createdAt, runs: [{sessionId, role, side, repo, status, startedAt, endedAt}]}
└── tasks.md, contracts/, decisions.md, schema.md, …  (unchanged)
```

**Runtime:** Bun + Hono. One process, serves API and static files on `http://localhost:7777`.

**Launch:** `bun run ui/server.ts` (add `ui` script to a new `package.json` at repo root).

## Data model

### Tasks — `tasks.md` (source of truth)

Entries follow the existing template (`- [ ] **YYYY-MM-DD** …`). The UI adds one machine-readable comment per task to tie it back to the sessions folder:

```md
- [ ] **2026-04-24** Add /users/me endpoint <!-- task-id: t_20260424_001 -->
  Contract: `contracts/users-me.md`
  Notes: needs email + roles
```

The UI never stores the task body anywhere else — re-parsing `tasks.md` is the only way to read tasks. Writes are atomic (read → modify → write whole file).

**Task ID format:** `t_YYYYMMDD_NNN` (date + daily sequence). Generated by the UI.

**Status mapping (tasks.md sections → task status):**
- `TODO — UNASSIGNED` *(new section, added by UI on first run)* → `todo`
- `TODO — FE` / `TODO — BE` *(existing; left in place for manually-authored handoffs; UI-dispatched tasks skip these)* → `todo`
- `DOING` → `doing`
- `BLOCKED` → `blocked`
- `DONE — not yet archived` → `done`

Moving a task between sections = changing status. The UI writes back in-place. **Task status** (above, reflected in tasks.md) is separate from **run status** (in meta.json, see below).

### Sessions — native Claude Code files

Sessions live at `~/.claude/projects/<slug>/*.jsonl` where slug is derived from repo path (Claude's existing convention: e.g. `D:\Edusoft\edusoft-lms-bridge` → `d--Edusoft-edusoft-lms-bridge`). The UI does not create these files — `claude` does.

Slug resolution helper: read the three repo absolute paths (bridge itself, and `../edusoft-lms`, `../edusoft-lms-api` as declared in BRIDGE.md Repos table), apply the Claude convention (replace `\` and `/` and `:` with `-`, collapse consecutive dashes).

### Meta mapping — `sessions/<task-id>/meta.json`

Bridges task ↔ Claude session IDs. The initial `coordinator` run entry is written by the UI at dispatch time; subsequent runs (coder, reviewer, fix cycles) are appended by the coordinator itself as it spawns subagents. Read by the UI to label sessions.

**Run status enum:** `queued` (entry created, process not yet started) · `running` (process alive) · `done` (exit 0) · `failed` (non-zero exit or crash) · `stale` (no heartbeat > 30 min, assumed dead).

```json
{
  "taskId": "t_20260424_001",
  "taskTitle": "Add /users/me endpoint",
  "createdAt": "2026-04-24T10:00:00Z",
  "runs": [
    { "sessionId": "abc-123", "role": "coordinator", "side": null,  "repo": "edusoft-lms-bridge", "status": "done",    "startedAt": "…", "endedAt": "…" },
    { "sessionId": "def-456", "role": "coder",       "side": "BE",  "repo": "edusoft-lms-api",    "status": "done",    "startedAt": "…", "endedAt": "…" },
    { "sessionId": "ghi-789", "role": "reviewer",    "side": "BE",  "repo": "edusoft-lms-api",    "status": "running", "startedAt": "…", "endedAt": null }
  ]
}
```

## Agent team

Three role prompts live in `agents/*.md` as plain-text templates with `{{PLACEHOLDER}}` markers (`{{TASK_ID}}`, `{{TASK_TITLE}}`, `{{TASK_BODY}}`, `{{SIDE}}`, `{{COORDINATOR_NOTES}}`).

- The **coordinator** prompt is assembled by `ui/lib/spawn.ts` (bridge backend) when the UI dispatches. Backend reads `agents/coordinator.md`, substitutes, and passes as `claude -p <prompt>` in the bridge cwd.
- The **coder** and **reviewer** prompts are assembled by the coordinator itself (which is a running `claude -p` session). The coordinator's prompt instructs it to `Read agents/coder.md`, substitute placeholders, and `Bash claude -p "<prompt>"` with `cwd=../edusoft-lms[-api]`. All string substitution happens via Read + Write + Bash inside the coordinator — `ui/lib/spawn.ts` is only responsible for the initial coordinator spawn.

### coordinator

- Runs in the bridge repo.
- Reads `BRIDGE.md`, `tasks.md` (only the target task), `contracts/`, `decisions.md`.
- Decides `side`: `FE`, `BE`, or `BOTH`.
- For each side: spawns a coder `claude -p` in `../edusoft-lms` or `../edusoft-lms-api` with the coder prompt + task context. Waits for completion (blocking, capture session ID).
- After coder finishes: spawns a reviewer `claude -p` in the same repo with the reviewer prompt + a pointer to the coder's session ID / diff.
- If reviewer reports `BLOCK`: coordinator spawns one follow-up coder run with the review notes, then one more reviewer run (single fix cycle, no infinite loop).
- Updates `meta.json` after each spawn (session ID, role, side, status).
- Updates `tasks.md` status at the end (`DOING` → `DONE — not yet archived`, or `BLOCKED` if stuck).
- Does NOT write code itself.

### coder

- Runs in the target sibling repo.
- Receives the task body + any referenced contract.
- Implements, commits on a topic branch (`task/<task-id>`). Does not push.
- Must write tests (per app's conventions).
- Exits with a summary and the branch name.

### reviewer

- Runs in the same repo as the coder just did.
- Checks out `task/<task-id>` branch.
- Reviews the diff against the task + contract + app conventions.
- Runs the app's test/lint commands.
- Outputs `APPROVE` (task ready) or `BLOCK: <reason>` (needs a fix cycle).

**How roles are tagged in sessions:** each spawn uses `claude -p` with a system-prompt prefix `[ROLE: coder|reviewer|coordinator] [TASK: t_...]`.

**Session ID capture strategy (two-layer):**
1. **Primary** — parse Claude Code's stdout for the session ID line it emits on start (format varies by version; spawn.ts matches a UUID-shaped token on the first N lines of stdout).
2. **Fallback** — if stdout parsing fails, the newest `.jsonl` file in the target project's `~/.claude/projects/<slug>/` directory (mtime-sorted) whose first message matches the `[ROLE: …] [TASK: …]` prefix is treated as the session for this run.

The coordinator (for coder/reviewer spawns) and `ui/lib/spawn.ts` (for the coordinator itself) both use the same strategy and both write back to `meta.json`.

## Backend API

Hono, JSON, no auth. Base: `http://localhost:7777/api`.

| Method | Path | Purpose |
|---|---|---|
| GET  | `/tasks` | list tasks (parsed from tasks.md) |
| POST | `/tasks` | create task → append to tasks.md + create `sessions/<task-id>/` |
| PATCH | `/tasks/:id` | edit title/body/status (rewrites tasks.md) |
| POST | `/tasks/:id/dispatch` | spawn coordinator `claude -p` in bridge repo (detached child, parent does not wait); returns `202` with `{taskId, coordinatorSessionId}` once the session ID is captured (usually < 2 s) |
| GET | `/tasks/:id/meta` | read `meta.json` |
| GET | `/sessions/:sessionId/tail?since=<n>` | read `.jsonl` from `since` bytes, return lines + new offset (poll loop from UI) |
| GET | `/repos` | read BRIDGE.md Repos table → `[{side, folder}]` with resolved sibling paths |

All paths are resolved as **siblings of the bridge folder** (per BRIDGE.md). Never hardcoded.

## Frontend UI

Single HTML page, three panes:

```
┌───────────────────────────────────────────────────────────────┐
│ Edusoft LMS — Bridge Coordinator            [+ New task]      │
├───────────────┬──────────────────────┬────────────────────────┤
│ TASKS         │ TASK DETAIL          │ SESSION VIEWER         │
│               │                      │                        │
│ • TODO (3)    │ t_20260424_001       │ coordinator ●running   │
│   - Add /me   │ Add /users/me        │ ───────────────────── │
│   - …         │                      │ [10:00] analyzing…    │
│ • DOING (1)   │ Body:                │ [10:01] side=BE       │
│   - …         │ [textarea]           │ [10:02] spawn coder   │
│ • BLOCKED(0)  │                      │                        │
│ • DONE (2)    │ Contract: users-me   │ coder ●done            │
│   - …         │                      │ ───────────────────── │
│               │ [Dispatch ▶]         │ [10:03] reading task  │
│               │                      │ …                      │
│               │ Runs:                │                        │
│               │ • coordinator ●done  │ reviewer ●running      │
│               │ • coder(BE)   ●done  │ ─────────────────────  │
│               │ • reviewer(BE) ●run  │ …                      │
└───────────────┴──────────────────────┴────────────────────────┘
```

- **Left pane:** groups by status (`tasks.md` sections). Click → loads detail.
- **Middle pane:** editable task fields + `Dispatch` button + list of runs from `meta.json`. Click a run → opens it in the right pane.
- **Right pane:** live tail of the selected session's `.jsonl`. Polls `/sessions/:id/tail` every 1 s with `since` byte-offset; appends new user/assistant/tool entries, rendered as collapsible blocks.

No framework. One `app.js`. State = plain object, re-renders panes on change. CSS grid layout, monospace for the session viewer.

## Data flow (happy path)

1. User clicks **+ New task**, fills title + body + optional contract link, saves.
   → `POST /tasks` → backend generates `t_20260424_001`, appends to `tasks.md` (TODO — FE/BE section left to coordinator; use a fourth section `TODO — UNASSIGNED` initially), creates `sessions/t_20260424_001/meta.json` skeleton.
2. User clicks **Dispatch**.
   → `POST /tasks/:id/dispatch` → backend spawns `claude -p` in bridge repo with coordinator prompt. Records coordinator `sessionId` in `meta.json`. Returns `202`.
3. Coordinator runs: reads bridge context → decides `BE` → moves task from `TODO — UNASSIGNED` to `DOING` in tasks.md → spawns coder `claude -p` in `../edusoft-lms-api`, updates `meta.json` with coder run.
4. Coder runs, commits on `task/t_20260424_001`, exits. Coordinator spawns reviewer.
5. Reviewer runs tests + reviews, exits `APPROVE`. Coordinator moves task to `DONE — not yet archived`, marks `meta.json` runs `done`, exits.
6. UI (polling meta + tailing JSONL) reflects every state transition within ~1 s.

## Error handling

- **Sibling repo missing:** `GET /repos` returns `exists: false`. UI disables Dispatch and shows a hint.
- **`claude` binary not on PATH:** dispatch returns `500` with clear message; UI shows a banner.
- **Coordinator crash mid-run:** `meta.json` keeps the last known state; UI marks coordinator run `failed`. No auto-retry — user clicks Dispatch again.
- **Reviewer returns BLOCK twice:** coordinator stops after one fix cycle; moves task to `BLOCKED` with the last reviewer message quoted in the task body. User decides next step.
- **tasks.md conflict (user edited by hand while UI was running):** writes are optimistic — read file, apply change, write. If mtime changed between read and write, return `409` and UI reloads. No merging.
- **Partial meta.json (process killed):** UI tolerates missing fields; treats missing `endedAt` + stale `startedAt` (> 30 min) as `stale`.

## Testing

- **`ui/lib/tasks.ts`:** unit tests for parse/serialize round-trip against a fixture `tasks.md`. Edge cases: empty sections, multi-line notes, existing `<!-- task-id: … -->` comments.
- **`ui/lib/sessions.ts`:** unit tests against fixture `.jsonl` files (byte-offset tail, incomplete last line).
- **`ui/lib/meta.ts`:** unit tests for read/write/append run.
- **`ui/lib/spawn.ts`:** mocked `child_process` tests — verifies args, cwd, env, and sessionId capture from stdout.
- **E2E (manual):** see "Launch & test" below.

No mocking of actual `claude` runs in CI — this is a local solo tool.

## Launch & test

Final step of the implementation: I start the UI, walk through one end-to-end task with the user.

1. `cd edusoft-lms-bridge && bun install && bun run ui/server.ts`
2. Open `http://localhost:7777`.
3. Create a trivial test task (e.g. "Add a `/health` endpoint returning `{ ok: true }`").
4. Click Dispatch. Observe:
   - coordinator session appears, classifies as BE,
   - coder session runs in `../edusoft-lms-api`,
   - reviewer session runs, approves,
   - task moves to DONE in the left pane,
   - `tasks.md` on disk reflects the same state.
5. If any step fails, debug before claiming done.

## Folder-name resolution

Folder names come from the Repos table in `BRIDGE.md`. `ui/lib/repos.ts` parses that table (markdown), resolves each sibling path as `path.resolve(bridgeRoot, "..", folderName)`, and exposes `{ FE: {folder, path, exists}, BE: {folder, path, exists} }`. No hardcoding.

## Out of scope (explicitly deferred)

- Editing sessions from the UI.
- Multiple parallel dispatches for the same task.
- Webhooks / remote triggers.
- Archiving `tasks.md` (user does this monthly, per bridge convention).
- Any code on the FE/BE side beyond what the agent team produces per task.
